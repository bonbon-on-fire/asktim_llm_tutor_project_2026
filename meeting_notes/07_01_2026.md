## Notes

### Second-round simulated-conversation results (324, with lecture context)

- Only 5 transcripts scored below 31, and of those 4 lost points on pedagogy — leaking or verifying answers.
- After hand-reviewing them, 3 were actually acceptable: e.g. the student gave the answer and the tutor confirmed it was right, which is good behavior. So those are fine.
- There was one genuine leak, where the tutor provided the answer by solving the equation for the student. Out of 324 total conversations that's less than 0.5%, which is perfectly fine.

### Cost and the shift to RAG

- Running the 324 conversations with lecture context cost ~$35, whereas the 108 conversations without lecture context cost <$1. Adding full lecture context to each exercise is therefore not scalable.
- We will shift to using RAG for context instead: run 324 more simulated conversations using RAG, grade them, and compare against the runs without RAG.
- For exercises that require lecture information to solve (e.g. exercise 2 in Supply Chain Design), one option is to pre-inject the necessary material so it's on hand without doing RAG or carrying the full lecture context.
- Now that we have the lecture architecture, another future-scaling option is to update the tutor prompt to cite the specific lectures the guidance is being pulled from.

### Student tracking and login

- Being able to follow students along their journey of using the tutor is an important tool.
- It should still be possible to opt out of adding an email/login, so we can't make it mandatory.
- With real human users now, it may be time to submit for IRB.
- Change student identity from prompting for an email to a username instead. This should incentivize students to create a login; it's completely anonymous and not connected to any student names.
- Change the wording in the email pop-up to talk about how signing up improves chat quality.
- Have the pop-up appear each time after 1 message, so it's a bit annoying (nudges sign-up).
- Change the pop-up header wording to something like "Help AskTIM improve […]".

### Future tutor improvements

- A/B testing responses like GPT does: provide two responses and have the student say which one is better.

### Discussion points for meeting with Inma

1. Simulated conversations show the tutor is working well on this course, with less than 0.5% of conversations showing answer leakage.
2. Ask whether the tutor can use future lecture context and teachings to help answer the current week's exercises, or whether it should fully avoid that.
3. Ask whether they want to use the tutor on practice problems along with the graded exercises — noting it gives students more learning opportunities and us more testing opportunities.

## Decisions

- Adding full lecture context to each exercise is not scalable (~$35 for 324 conversations vs <$1 without), so shift to using RAG for course context.
- The single answer-leak out of 324 conversations (<0.5%) is acceptable; the tutor is working well on this course.
- For exercises that require lecture information to solve (e.g. exercise 2 in Supply Chain Design), pre-inject the necessary material rather than relying on RAG or the full lecture context.
- Change student identity from an email prompt to an anonymous username to incentivize login, while keeping it optional (not mandatory).
- Rework the sign-up pop-up to nudge more sign-ups: reframe the wording around improving chat quality, change the header, and show it after every message.
